做小程序遇到的问题

经过自己的努力，自己的小程序终于等到了更新。虽然，在这个过程当中自己遇到了很多问题，但是等到更新的那一霎那，自己感觉太爽了。把自己原来所设想的功能做了出来。一方面自己需要整理自己所遇到的问题，另一方面自己还需要持续更新小程序。

所遇到的问题：get请求，上传问题，解密（get请求，），小程序端登录（后台），选择器的值如何给form表单，分类浏览（后台的逻辑，前台的请求。）

get请求的要点：1，请求的格式，URL地址，返回的数据需要获取到。2，需要在后台写好articles.json的接口

给服务器端上传问题。在第一次上传的时候自己忘记了执行heroku run rake db:migrate。第二次上传遇到的问题，数据格式不正确。因为自己在判断方法后面多写了一个？，导致语法错误。按照提示，自己总是会不由自主地向下看代码，而真正的错误是出现在提示上面。所以自己以后遇到错误，一定要看代码的上下文。

解密，首先需要在小程序端向后台服务器发起请求，（携带的的参数有code，vi，密文。在这里不需要对密文进行转义。如果进行转义的话，那么就不能解密成功。用Base64解密之后的内容，一般在最后面有一个=。自己在这个问题上面花费了很多时间。）

后台服务器接到请求之后，先把参数存起来，然后拼接出一个连接（向微信服务器发起请求），请求成功，微信服务器返回（openid，sessionkey。利用openId创建wx_user的一条记录，sessionkey用作解密的参数）。然后后台服务器进行解密（解密需要用到的参数有：密文，appid，vi，sessionkey，解密的方法写到模型当中，解密的时候控制器直接调用解密方法）。解密之后，就可以获取到用户的信息，把有用的参数存到数据库当中（解密之后有一个openId与openid进行比较，如果相同，更新wx_user表的其他字段）。在wx_user的模型当中，写入一个after_create，创建一条user记录。wx_user和user表的关系是一对一。最后，在wx_user控制器当中写入返回给小程序段的数据。（这里也需要注意格式，直接返回json数据即可，实际上只需要返回wx_user_id即可）

自己原来思考的问题是：想着让小程序端如何登录？实际上，自己想解决的问题是：在小程序端写一篇文章，然后这篇文章必须要属于一个用户。这两个问题的描述，第个问题描述的更加具体，相对而言也更加好。解决的的方法是，把后台服务器端返回的wx_user_id存到storage当中（同步存储）。然后提交文章的时候，把storage当中的wx_user_id读取出来，post提交请求的时候携带上wx_user_id这个参数。（拼接成想要的一个连接，不过格式需要注意。自己原来完全不知道如何拼接，通过阅读其他的代码，不断地尝试才找到答案。自己需要记住格式，自己这一次拼接的url地址，只有一个变化的参数，自己需要弄清楚两个两个变化的参数如何拼接。答案在豆瓣电影的小程序当中。）到了后台服务器，先判断有没有wx_user_id这个参数，如果有的话，通过wx_user_id这条记录，然后在通过这条记录找到想对应的user记录，然后把user.id赋值给article.user_id。如果没有这个参数的话，直接把当前用户的id赋值给article.user_id。（这种情况属于后台创建文章的情况。）

选择器的值如何给form表单。自己需要隐藏掉携带值。写入需要提交的name，例如article[status]。

分类浏览。实现的功能是：按全部时可以查看全部的文章，按草稿时可以查看草稿类的文章，按发布时可以查看已经发布的文章。整体是一个滑动的效果。不同的按钮会传输不同的index值。（索引值）自己可以通过索引值判断出是哪个按钮，然后在提交相应的post请求。提交请求之后，分析携带的参数，给予不同的处理。携带的的参数有三种情况：只要wx_user_id，wx_user_id和status=0，wx_user_id和status=1。后台对于这些不同的情况，利用过滤条件返回出不同的值。

下一个星期需要更新的功能：上传图片，详情页面，删除和编辑功能，整个UI。主要先完成删除，编辑和详情功能。

更完整的代码，请参考：https://github.com/xueshaojie/xiaochengxu，wx_blog是小程序端的代码。
